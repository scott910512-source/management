01 — Repository

레포지토리

프로젝트 하나 = 레포지토리 하나.
모든 파일과 전체 변경 이력이 여기에 저장됩니다.

📂

프로젝트 저장소

코드, 이미지, 설정파일 등 프로젝트에 필요한 모든 것을 담는 공간

📜

이력 관리

언제 무엇을 바꿨는지 전부 기록. 언제든 이전 버전으로 되돌릴 수 있음

☁️

원격 백업

GitHub에 올려두면 어디서든 접근 가능하고 날아갈 걱정 없음

02 — Branch

브랜치

현재 코드를 그대로 복사한 독립 작업공간.
main은 항상 완성된 코드만 두고, 새 기능은 브랜치를 따서 작업합니다. 

# 브랜치 만들고 이동
git checkout -b feature/로그인

# 현재 브랜치 확인
git branch
이름용도
feature/새 기능 개발
fix/버그 수정
hotfix/긴급 수정
docs/문서 작업
refactor/코드 구조 정리
핵심: main은 건드리지 않는다. 항상 브랜치에서 작업한다.

03 — Commit

작업 & 커밋

파일을 수정한 뒤 커밋으로 저장합니다.
커밋은 "이 시점을 저장해 둔 스냅샷" — 나중에 이 시점으로 되돌릴 수 있습니다. 

# 변경 파일 스테이징
git add .

# 커밋 — 무엇을 했는지 한 줄로
git commit -m "feat: 로그인 화면 추가"

# GitHub에 올리기
git push origin feature/로그인
메시지 prefix언제 쓰나
feat:새 기능 추가
fix:버그 수정
style:디자인·레이아웃 변경
refactor:코드 구조 변경 (기능 변화 없음)
docs:문서·주석 변경
remove:파일·코드 삭제

04 — Merge

머지

브랜치에서 완성된 작업을 main에 합치는 것.
GitHub에서 Pull Request(PR) 를 만들고 버튼 하나로 끝납니다.

1

GitHub에서 PR 생성

브랜치 push 후 GitHub 페이지에서 "Compare & pull request" 클릭

2

변경 내용 확인

어떤 파일이 바뀌었는지 diff 보기. 혼자라면 그냥 쭉 훑고 넘어가도 됨

3

Merge pull request

버튼 클릭 → main에 반영 완료

4

브랜치 삭제 & 로컬 업데이트

머지된 브랜치는 Delete branch로 정리. 로컬도 git pull origin main 으로 최신화

05 — Collaboration

팀으로 쓸 때 추가되는 것

기본 흐름은 혼자 쓸 때와 같습니다.
추가되는 건 딱 두 가지 — 코드 리뷰와 충돌 처리.

👀

코드 리뷰

PR을 만들면 팀원이 코드를 검토하고 Approve. 이후 머지합니다.

⚡️

빠른 머지

브랜치를 오래 유지할수록 충돌 위험이 높아집니다. 작게 쪼개서 빨리 머지하세요.

충돌(Conflict)이 생겼을 때

같은 파일의 같은 줄을 두 브랜치에서 동시에 수정하면 발생합니다.

<<<<<<< feature/내-브랜치
빨간색 버튼
=======
파란색 버튼
>>>>>>> main
1

충돌 표시 구간 찾기

<<< === >>> 로 표시된 부분을 VS Code 등에서 열기

2

원하는 내용만 남기고 저장

표시 기호를 모두 삭제하고, 최종으로 쓸 코드만 남김

3

다시 커밋

git add . → git commit -m "fix: 충돌 해결"

충돌 예방: 팀원마다 다른 파일을 담당하고, 브랜치를 오래 끌지 않으면 대부분 피할 수 있습니다.
