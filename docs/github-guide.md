# GitHub 브랜치 / 작업 / 머지 사용 가이드

---

## 1. 브랜치(Branch)란?

브랜치는 **전체 파일을 그대로 복사한 독립적인 작업공간**입니다.

- 브랜치를 새로 만들면 → 현재 전체 파일을 기준으로 시작
- 브랜치 안에서 작업한 내용은 **다른 브랜치에 영향을 주지 않음**
- `main` 브랜치는 항상 **완성된 안정적인 코드**만 유지

```
main ──────────────────────────────────▶
         │
         └─── feature/login-ui ──▶ (로그인 화면 작업)
         │
         └─── feature/canister-api ──▶ (API 서버 작업)
```

---

## 2. 브랜치 생성

```bash
# main 기준으로 새 브랜치 생성 후 이동
git checkout -b feature/기능명

# 예시
git checkout -b feature/login-ui
git checkout -b fix/dashboard-bug
```

### 브랜치 네이밍 규칙 (권장)

| prefix | 용도 |
|--------|------|
| `feature/` | 새 기능 개발 |
| `fix/` | 버그 수정 |
| `hotfix/` | 긴급 수정 |
| `docs/` | 문서 작업 |
| `refactor/` | 코드 정리 |

---

## 3. 작업(Commit)

브랜치에서 파일을 수정한 뒤 **커밋(commit)** 으로 저장합니다.
커밋은 특정 시점의 스냅샷(버전)입니다.

```bash
# 수정된 파일 스테이징
git add 파일명
git add .          # 전체 변경사항

# 커밋 (메시지는 무엇을 했는지 명확하게)
git commit -m "feat: 로그인 화면 디자인 추가"
git commit -m "fix: 대시보드 버그 수정"

# 원격 저장소(GitHub)에 푸시
git push origin feature/login-ui
```

### 커밋 메시지 규칙 (권장)

| prefix | 용도 |
|--------|------|
| `feat:` | 새 기능 |
| `fix:` | 버그 수정 |
| `docs:` | 문서 변경 |
| `style:` | 디자인/포맷 변경 |
| `refactor:` | 코드 구조 변경 |
| `remove:` | 파일/코드 삭제 |

---

## 4. 머지(Merge)란?

브랜치에서 완성된 작업을 **main 브랜치에 반영**하는 과정입니다.

```
feature/login-ui ──── (작업완료) ────┐
                                     ▼
main ────────────────────────────── merge ──▶
```

### GitHub에서 머지하는 방법

1. GitHub에서 **Pull Request(PR)** 생성
2. 변경 내용 확인
3. **Merge pull request** 클릭
4. 머지 완료 후 브랜치 삭제 (선택)

---

## 5. 충돌(Conflict) 처리

**같은 파일의 같은 줄**을 두 브랜치에서 동시에 수정하면 충돌이 발생합니다.

```
<<<<<<< feature/브랜치1
빨간색 버튼
=======
파란색 버튼
>>>>>>> feature/브랜치2
```

해결 방법:
1. 충돌 표시(`<<<<<<<`, `=======`, `>>>>>>>`) 부분을 직접 수정
2. 원하는 내용만 남기고 저장
3. 다시 커밋 후 머지

충돌을 줄이려면:
- 서로 **다른 파일**을 담당하도록 작업 분리
- 브랜치를 **오래 유지하지 않고 빠르게 머지**

---

## 6. 전체 흐름 요약

```
1. main에서 브랜치 생성
   git checkout -b feature/기능명

2. 파일 수정 후 커밋
   git add .
   git commit -m "feat: 기능 설명"

3. GitHub에 푸시
   git push origin feature/기능명

4. GitHub에서 Pull Request 생성

5. 확인 후 main에 머지

6. 브랜치 삭제
```

---

## 7. 핵심 개념 정리

| 개념 | 설명 |
|------|------|
| 브랜치 | 전체 파일 기반의 독립 작업공간 |
| 커밋 | 특정 시점의 코드 스냅샷(버전) |
| 푸시 | 로컬 작업을 GitHub에 업로드 |
| PR | 브랜치를 main에 반영 요청 |
| 머지 | PR을 승인해 main에 반영 |
| 충돌 | 같은 파일 같은 줄을 동시 수정할 때 발생 |
