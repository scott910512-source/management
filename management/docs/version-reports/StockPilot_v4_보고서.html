버전 컨셉 · 기술 배경

1공장·2공장 데이터를 폴더 단위로 완전 분리하고, X-Plant HTTP 헤더로 공장 컨텍스트 전달 — 서버 시작 시 자동 백업

공장별 데이터 분리 구조

// server/data/ 폴더 구조
server/data/
├── 1공장/
│   ├── items.csv
│   ├── raw_materials.csv
│   ├── transactions.csv
│   └── ...
├── 2공장/
│   └── ...  // 완전 독립 파일셋
└── users.csv  // 공통 — 공장 구분 없음

X-Plant 헤더 처리

// client: 한글 공장명 → URL 인코딩
headers: { 'X-Plant': encodeURIComponent(currentPlant) }

// server/src/middleware/plant.js — 복호화 + 경로 생성
const resolvePlant = (req, res, next) => {
  const plant = decodeURIComponent(req.headers['x-plant'] || '');
  if (!plant) return next(badRequest('공장을 선택하세요.'));
  req.plant = plant;  // 이후 readTable/mutate에서 폴더 경로로 사용
  next();
};

// store.js: plant 인자 → 파일 경로 결정
const filePath = (table, plant) =>
  path.join(DATA_DIR, plant || '', table + '.csv');

서버 시작 시 자동 백업

// server/src/index.js — 서버 기동 시 스냅샷
async function createBackup() {
  const stamp = new Date().toISOString().replace(/[:.]/g, '-');
  const dest = path.join(BACKUP_DIR, stamp);
  await fs.cp(DATA_DIR, dest, { recursive: true });
  // 최근 20개만 보관 — 오래된 것 자동 삭제
  const backups = fs.readdirSync(BACKUP_DIR).sort();
  if (backups.length > 20) {
    backups.slice(0, backups.length - 20).forEach(b =>
      fs.rmSync(path.join(BACKUP_DIR, b), { recursive: true }));
  }
}

주요 기능 내용

실제 구현된 기능과 기술적 처리 방식

공장별 데이터 분리
server/data/1공장/, server/data/2공장/ 폴더로 완전 분리. users.csv만 공통(공장 인자 없이 접근).
X-Plant 헤더
모든 API 요청에 X-Plant: encodeURIComponent(공장명) 전송. 서버 미들웨어 resolvePlant에서 디코딩 → req.plant로 주입.
공장 선택기
사이드바 드롭다운 — 로그인 응답의 plants[] 배열에서 권한 있는 공장만 표시. 선택 시 localStorage에 저장.
자동 백업
서버 시작(index.js)마다 createBackup() 실행. 타임스탬프 폴더로 스냅샷. 최신 20개만 보관.
데이터 보존 정책
배포/업데이트 시 server/data/ 폴더를 절대 덮어쓰지 않음. 코드와 데이터 폴더 분리로 안전 업데이트.
권한 기반 공장 노출
로그인 응답에 plants: ["1공장"] 같이 허용 공장 목록 포함. UI에서 허용 공장만 선택 가능.

기능 강화 · 인터락 (안전장치)

실수·사고·무단 접근을 막는 서버 레벨 안전장치

데이터 보존
업데이트 스크립트에서 server/data/ 폴더는 건드리지 않는 규칙. 배포가 기존 자료를 덮어쓰지 못함.
자동 백업 안전망
서버 켤 때마다 백업 → 잘못 저장 후에도 이전 스냅샷으로 복구 가능. 20개 보관 한도로 디스크 관리.
권한 기반 차단
서버에서 req.user.plants.includes(req.plant) 검사. 허용 안 된 공장 요청은 403 반환.
공통 데이터 분리
users 등 전체 공통 데이터는 plant 인자 없이 루트 data 폴더 사용. 공장 컨텍스트 오염 방지.

버그 수정 · 원인 분석

발생 원인과 해결 방법

한글 헤더 깨짐
HTTP 헤더에 한글 직접 포함 시 일부 브라우저·서버에서 깨짐 → encodeURIComponent ↔ decodeURIComponent 쌍으로 해결.
공통 데이터 공장 혼용
초기엔 users 테이블에도 plant 인자 전달 → 공장별 users.csv가 생성되는 문제 → 전역 테이블 목록 GLOBAL_TABLES로 분리.
권한 빈값 허용
역할 체크에서 빈 문자열이 전체 허용으로 통과되는 버그 → 명시적 허용 목록 배열로 교체.

학습 · 기술 포인트

이 버전에서 익힌 핵심 기술 개념

멀티테넌시 설계
폴더 경로를 런타임 파라미터로 변경하는 방식으로 테넌트(공장)별 데이터 격리. DB 파티션 없이 파일 시스템으로 구현.
HTTP 헤더 인코딩
RFC 7230: HTTP 헤더 값은 Latin-1 범위여야 함. 한글은 percent-encoding 필수. 클라-서버 쌍으로 처리해야 완성.
백업 전략
스냅샷(전체 복사) vs. 증분 백업 중 단순성으로 스냅샷 채택. 보관 한도(20개)로 무제한 증가 방지.
