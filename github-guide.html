저장소는 프로젝트, 브랜치는 소작업 공간, 머지로 취합 — 협업의 핵심 흐름을 한눈에.

Branch Commit Pull Request Merge

01 — Branch

브랜치란 무엇인가

전체 파일을 그대로 복사한 독립적인 작업공간. 내 작업이 다른 사람 코드에 영향을 주지 않습니다.

🌿

독립 작업공간

브랜치에서 한 작업은 다른 브랜치에 전혀 영향을 주지 않습니다.

🔒

main은 항상 안정

main 브랜치에는 완성되고 검증된 코드만 유지합니다.

⚡️

병렬 개발 가능

여러 기능을 동시에 각자 다른 브랜치에서 개발할 수 있습니다.

main
feature/login-ui
로그인 화면 작업
feature/canister-api
API 서버 작업
terminal — 브랜치 생성 
# main 기준으로 새 브랜치 생성 후 이동
git checkout -b feature/기능명

# 예시
git checkout -b feature/login-ui
git checkout -b fix/dashboard-bug
Prefix용도예시
feature/새 기능 개발feature/login-ui
fix/버그 수정fix/dashboard-bug
hotfix/긴급 수정hotfix/payment-crash
docs/문서 작업docs/api-readme
refactor/코드 정리refactor/auth-module

02 — Commit

작업과 커밋

커밋은 특정 시점의 코드 스냅샷. 의미 있는 단위로 나눠 저장하세요.

terminal — 커밋 & 푸시 
# 수정된 파일 스테이징
git add 파일명
git add .          # 전체 변경사항

# 커밋 (무엇을 했는지 명확하게)
git commit -m "feat: 로그인 화면 디자인 추가"
git commit -m "fix: 대시보드 버그 수정"

# 원격 저장소(GitHub)에 푸시
git push origin feature/login-ui
Prefix용도예시 메시지
feat:새 기능feat: 로그인 화면 디자인 추가
fix:버그 수정fix: 대시보드 렌더링 오류 수정
docs:문서 변경docs: API 사용법 README 업데이트
style:디자인·포맷style: 버튼 색상 및 간격 수정
refactor:코드 구조 변경refactor: 인증 모듈 분리
remove:삭제remove: 사용하지 않는 레거시 코드 제거

03 — Merge

머지 & Pull Request

완성된 작업을 main 브랜치에 반영하는 과정. PR을 통해 코드 리뷰 후 머지합니다.

1

브랜치 작업 완료 후 GitHub에 Push

git push origin feature/login-ui 로 작업 내용을 GitHub에 올립니다.

2

Pull Request 생성

GitHub 저장소 페이지에서 Compare & pull request 버튼을 클릭해 PR을 만듭니다.

3

변경 내용 확인 & 리뷰

팀원이 코드를 검토하고, 문제가 없으면 Approve(승인)합니다.

4

Merge pull request 클릭

승인 후 Merge pull request 버튼을 눌러 main에 반영합니다.

5

브랜치 삭제

머지된 브랜치는 Delete branch로 정리합니다. main은 항상 최신 상태를 유지합니다.

04 — Conflict

충돌(Conflict) 처리

같은 파일의 같은 줄을 두 브랜치에서 동시에 수정하면 충돌이 발생합니다.

<<<<<<< feature/브랜치1
빨간색 버튼
=======
파란색 버튼
>>>>>>> feature/브랜치2
✏️

1. 직접 수정

충돌 표시(<<<, ===, >>>) 구간을 찾아 원하는 내용만 남기고 나머지를 삭제합니다.

💾

2. 저장 후 커밋

수정한 파일을 저장하고 git add . → git commit 으로 충돌 해결을 기록합니다.

📂

충돌 예방 팁 1

팀원마다 다른 파일을 담당하도록 작업을 분리하면 충돌 자체를 피할 수 있습니다.

⚡️

충돌 예방 팁 2

브랜치를 오래 유지하지 말고 빠르게 머지하세요. 브랜치가 오래될수록 충돌 가능성이 높아집니다.

05 — Full Flow

전체 흐름 한눈에

처음부터 끝까지 한 사이클의 실제 명령어 순서입니다.

terminal — 전체 워크플로우 
# ① main에서 브랜치 생성
git checkout -b feature/기능명

# ② 파일 수정 후 커밋
git add .
git commit -m "feat: 기능 설명"

# ③ GitHub에 푸시
git push origin feature/기능명

# ④ GitHub에서 Pull Request 생성 → 리뷰 → Merge

# ⑤ 로컬 main 업데이트 & 브랜치 정리
git checkout main
git pull origin main
git branch -d feature/기능명

06 — Summary

핵심 개념 정리

6가지 개념만 기억하면 GitHub 협업이 보입니다.

01
브랜치
전체 파일 기반의 독립 작업공간. main에서 분기해 기능별로 관리합니다.
02
커밋
특정 시점의 코드 스냅샷(버전). 의미 있는 단위로 나눠 저장합니다.
03
푸시
로컬 작업 내용을 GitHub 원격 저장소에 업로드합니다.
04
PR
브랜치를 main에 반영 요청. 코드 리뷰의 공식 채널입니다.
05
머지
PR을 승인해 브랜치 작업을 main에 통합합니다.
06
충돌
같은 파일 같은 줄을 동시 수정할 때 발생. 직접 수정 후 커밋으로 해결합니다.

GitHub 브랜치 가이드

저장소 → 브랜치 → 커밋 → PR → 머지 — 이 흐름 하나면 충분합니다.
