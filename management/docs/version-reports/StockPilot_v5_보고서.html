버전 컨셉 · 기술 배경

관리자·공장관리자·팀관리자(조회전용)·사용자 역할별 권한을 미들웨어로 강제 — 공장 간 데이터 완전 격리

역할 정의 및 권한 매핑

// 역할 5종: admin · admin1 · admin2 · team · user
const ROLE_PLANTS = {
  admin:  null,         // 전체 공장 접근
  admin1: ['1공장'],   // 1공장만
  admin2: ['2공장'],   // 2공장만
  team:   null,         // 전체 조회 가능 (읽기 전용)
  user:   'own-plant', // 본인 배속 공장만
};

미들웨어 3단계 방어

// server/src/middleware/auth.js
const requireAuth = (req, res, next) => {
  if (!req.session?.user) return res.status(401).json({ error: '로그인 필요' });
  next();
};

const requireAdmin = (req, res, next) => {
  if (!['admin','admin1','admin2'].includes(req.session.user.role))
    return res.status(403).json({ error: '관리자 전용' });
  next();
};

const requireWrite = (req, res, next) => {
  if (req.session.user.role === 'team')  // 조회전용 전역 차단
    return res.status(403).json({ error: '조회 전용 계정' });
  next();
};

// 공장 접근 권한 검사 (resolvePlant 내부)
const allowed = ROLE_PLANTS[role];
if (allowed && !allowed.includes(plant))
  return res.status(403).json({ error: '접근 권한 없는 공장' });

주요 기능 내용

실제 구현된 기능과 기술적 처리 방식

역할 5종
admin(전체), admin1(1공장), admin2(2공장), team(전체조회), user(배속공장). 로그인 응답에 role·plants 포함.
조회전용 전역 차단
requireWrite 미들웨어: team 역할이 POST·PATCH·DELETE 요청 시 모두 403. 라우터별 적용이 아닌 공통 미들웨어로 일괄 처리.
공장 간 격리
resolvePlant: 요청한 공장이 역할 허용 목록에 없으면 403. admin1이 2공장 데이터에 접근하면 차단.
쓰기 버튼 숨김
클라이언트에서 canWrite 체크로 수정·삭제 버튼 숨김. 서버 차단과 이중 방어.
역할 뱃지·라벨
사이드바에 현재 역할 표시. 조회전용(team)은 주황색 "조회 전용" 뱃지로 시각 구분.
공장 목록 응답
로그인 POST /auth/login 응답에 plants: ["1공장"] 포함. 프론트가 이 목록으로 선택기 구성.

기능 강화 · 인터락 (안전장치)

실수·사고·무단 접근을 막는 서버 레벨 안전장치

조회전용 전역 차단
requireWrite가 모든 수정 라우터에 적용. team 역할은 어떤 API로도 데이터를 변경할 수 없음.
공장 경계 강제
서버에서 역할별 허용 공장 목록과 요청 공장을 비교. 클라이언트 조작으로 우회 불가.
이중 방어
버튼 숨김(UX)과 미들웨어 차단(보안) 두 층으로 운영. 버튼이 보여도 API 차단, API를 직접 호출해도 차단.

학습 · 기술 포인트

이 버전에서 익힌 핵심 기술 개념

RBAC 패턴
역할별 허용 액션을 미들웨어 함수로 분리. 라우터에 router.post("/", requireWrite, requireAdmin, handler) 체인 구성.
이중 방어 원칙
UI 숨김은 UX, API 차단은 보안. 둘 중 하나만으론 부족 — 항상 서버가 최종 권한 결정자.
최소 권한 원칙
역할마다 필요한 최소한의 공장·액션만 허용. 명시적 허용 목록 방식(whitelist) 채택.
